# AI Orchestration Skill

**Personal skill for orchestrating AI agents strategically across complex, multi-step work.**

This skill institutionalizes patterns from analyzing 382 agent transcripts and identifying 106+ examples of strategic AI behavior management. Use this when planning complex work, debugging stubborn issues, or coordinating multiple AI agents.

---

## When to Use This Skill

**Triggers:**
- Complex, multi-step work requiring coordination
- Debugging sessions that have run >3 iterations without progress
- Work requiring multiple AI agents with specialized tasks
- Quality-critical deliverables (design systems, architecture decisions, public-facing work)
- When you notice AI giving generic or hallucinated responses

**Do NOT use for:**
- Simple, single-file edits
- Quick refactors with clear scope
- Routine maintenance tasks

---

## Core Principles (Evidence-Based)

From the analysis of 382 transcripts, these 6 principles drive 3-10x better outcomes:

### 1. Evidence Over Assumption
**Finding:** Users providing screenshots, logs, source documents get 3-5x better results with fewer iterations.

**Practice:**
- Upload visual evidence early (screenshots, Figma URLs, deployed sites)
- Tag specific files using @ mentions before asking questions
- Provide source documents (PDFs, decision memos, specs) as ground truth
- Require AI to read files before proposing solutions

**Example:**
> "here's screenshot of the Scaleway Console quota display to show the AI exactly what UI pattern to reference"

---

### 2. Constraints Over Freedom
**Finding:** Users who set clear constraints ("do NOT edit this file", "read technical constraints first") prevent hallucination and generic responses.

**Practice:**
- Set "Do NOT" boundaries as explicitly as "Do" instructions
- Provide technical constraints as design drivers, not limitations
- Define file/folder boundaries before AI starts work
- Require justification for decisions that impact other systems

**Example:**
> "Read the technical constraints mentioned in @decision-memo.pdf before proposing features"

---

### 3. Systems Over Solutions
**Finding:** Users who think in systems (z-index hierarchies, multi-agent workflows, feedback loops) create scalable, maintainable outcomes.

**Practice:**
- Ask for architecture diagrams before implementation
- Connect frontend ↔ backend ↔ infra impact in problem framing
- Design workflows with feedback loops, not linear steps
- Reference patterns across sessions (cross-project thinking)

**Example:**
```
z-0:  Canvas (3D model)
z-10: FlowTextOverlay
z-20: LoadProgressBar
z-30: PaginationBar
z-40: MobileTopNav
z-50: Reader controls
z-60: InfoPanel / Mobile banner
```

---

### 4. Checkpoints Over Leaps
**Finding:** Users who run tight correction loops with evidence at each step debug 10x faster.

**Practice:**
- Provide runtime evidence (logs, screenshots, measurements) at each iteration
- Use instrumentation as proof: "Issue reproduced, please proceed"
- Treat production deploys as verification checkpoints
- Run tight loops: Error → Evidence → Fix → Verify → Repeat

**Pattern:**
```
User: [Pastes 55-line build log with exact error]
AI: [Proposes fix]
User: [Uploads screenshot showing fix failed]
AI: [Adds runtime instrumentation]
User: "Issue reproduced, please proceed"
AI: [Collects measurements, discovers root cause]
```

---

### 5. Craft Over Speed
**Finding:** Users who insist on naming conventions, documentation, and enforcement mechanisms build professional-grade systems.

**Practice:**
- Reject one-off fixes; demand systematic approaches
- Anchor decisions in industry standards and best practices
- Run multi-iteration correction loops for completeness
- Create constants, documentation, and enforcement (ESLint rules)

**Example:**
> "Based on industry best practices for modern web development repositories... Unix/Linux convention - Most system directories are lowercase (`/bin`, `/usr`, `/home`)... Git-friendly - Avoids case-sensitivity issues across macOS/Windows/Linux"

---

### 6. Meta-Analysis Over Routine
**Finding:** Users who study their own AI usage patterns and create skills/templates from successful patterns compound their effectiveness.

**Practice:**
- Create skill documents from successful patterns
- Build reusable templates and checklists
- Document architecture decisions
- Create enforcement mechanisms (ESLint rules, CI checks)

## Strategic Techniques (Tactical Playbook)

### Technique 1: Probabilistic UI
**When:** UI/UX work where "clean and modern" is too vague

**How:**
1. Upload screenshot of reference UI pattern
2. Provide Figma/Notion URL as live reference
3. Tag specific component files with @ mentions
4. Specify exact visual behavior (spacing, states, animations)

**Impact:** Prevents generic responses, forces specificity

---

### Technique 2: Evidence Gates
**When:** Complex feature work with technical constraints

**How:**
1. List files AI must read before proposing solutions
2. Provide decision memos or specs as ground truth
3. Require AI to summarize constraints in its own words
4. Only proceed after AI demonstrates understanding

**Impact:** Prevents hallucination, ensures factual accuracy

---

### Technique 3: Multi-Agent Parallelization
**When:** Work with 5+ distinct components or tasks

**How:**
1. Break work into independent components
2. Deploy parallel agents with specialized tasks:
   - Agent 1: Research (read docs, competitive analysis)
   - Agent 2: Implementation (component A, B, C)
   - Agent 3: Verification (quality checks, testing)
   - Agent 4: Documentation (guides, diagrams)
3. Synthesize outputs into cohesive whole

**Impact:** 4x speedup, better quality through specialization

**Example:**
> "use multiagents" → 5 agents processing 23 vocabulary entries in parallel
> Sequential: 46 minutes → Parallel: 10-12 minutes

---

### Technique 4: Checkpoint Debugging
**When:** Debugging has run >3 iterations without resolution

**How:**
1. Paste full error log (don't summarize)
2. Upload screenshot of broken state
3. Add runtime instrumentation (console.logs, DevTools measurements)
4. Say "Issue reproduced, please proceed" after each iteration
5. Collect measurements before proposing next hypothesis

**Impact:** Reduces debugging time from hours to minutes

---

### Technique 5: Craft Enforcement
**When:** AI proposes "good enough" solution that violates patterns

**How:**
1. Reject solution with specific reason (naming, consistency, best practice)
2. Demand systematic approach (not one-off fix)
3. Require industry standard justification (not "it's common")
4. Create enforcement mechanism (constant, ESLint rule, documentation)

**Impact:** Creates maintainable, scalable systems

---

### Technique 6: Validation Checkpoints
**When:** Quality-critical deliverables (design systems, public APIs)

**How:**
1. Build self-evaluation into AI workflow before presenting output
2. Require AI to run validation tests:
   - The swap test (would you swap this for competitor's version?)
   - The squint test (does it hold up at a glance?)
   - The signature test (does it have a distinct point of view?)
   - The token test (does it use design tokens consistently?)
3. Require "WHY" for each major decision

**Impact:** Catches generic patterns before user has to reject them

## Multi-Stage Workflow Template

**Use this structure for complex, multi-step work:**

### Phase 1: Research (Evidence Gathering)
- Read relevant files, docs, specs
- Understand context and constraints
- Identify dependencies and cross-system impact
- **Exit criterion:** AI can summarize problem in own words

### Phase 2: Planning (Strategy Before Tactics)
- Create approach document or plan
- Get user approval before implementation
- Define success metrics and validation criteria
- **Exit criterion:** User explicitly approves plan

### Phase 3: Implementation (Execute with Multi-Agents)
- Deploy parallel agents for specialized tasks
- Component-by-component execution
- Maintain consistency across components
- **Exit criterion:** All components implemented and tested

### Phase 4: Verification (Test, Verify, Document)
- Run validation checkpoints
- Create documentation and guides
- Extract patterns for future reuse
- **Exit criterion:** Work is merge-ready and documented

## Quality Metrics

**Use these to evaluate AI output quality:**

### The Swap Test
Would you swap this for a competitor's version? If yes, it's generic.

### The Squint Test
Does it hold up at a glance? Or does it fall apart when you step back?

### The Signature Test
Does it have a distinct point of view? Or could it be from any AI?

### The Token Test
Does it use design tokens and system patterns consistently?

### The Why Test
Can the AI explain WHY for each major decision? Or does it default to "it's common"?

## Common Anti-Patterns (Avoid These)

### ❌ Generic Solution Proposals
**Symptom:** AI proposes "clean and modern" solution without specificity

**Fix:** Upload reference screenshot, demand exact visual behavior

### ❌ Hallucinated Constraints
**Symptom:** AI invents technical limitations or API capabilities

**Fix:** Require file reading first, provide source documents as ground truth

### ❌ One-Off Fixes
**Symptom:** AI solves immediate problem but violates naming/patterns

**Fix:** Demand systematic approach, create enforcement mechanism

### ❌ Long Debugging Sessions
**Symptom:** >3 iterations without progress, AI guessing hypotheses

**Fix:** Add instrumentation, collect measurements, provide evidence

### ❌ Missing Validation
**Symptom:** AI presents output without justifying decisions

**Fix:** Require "WHY" for each decision, run validation checkpoints

## Example Workflows

### Workflow 1: Complex Feature Implementation

```
1. User uploads:
   - Screenshot of reference UI (@quota-display.png)
   - Technical constraints (@decision-memo.pdf)
   - Files to edit (@components/QuotaTable.tsx)

2. AI reads all files, summarizes constraints

3. User deploys multi-agents:
   - Agent 1: Research (read Scaleway design system docs)
   - Agent 2: Implementation (QuotaTable component)
   - Agent 3: Implementation (QuotaCard component)
   - Agent 4: Verification (accessibility, responsive checks)

4. Each agent runs validation checkpoints before presenting output

5. User synthesizes outputs, runs final validation tests
```

**Result:** 4x faster than sequential, higher quality through specialization

